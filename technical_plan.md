# AI StoryCards App — Technical Architecture & Implementation Plan

## 1. Product Overview

StoryCards is a mobile application that allows users to **experience books, movies, anime, and TV shows through AI-generated visual story cards**.

Each story is broken into a sequence of cards. Every card contains:

- short narrative text (1–2 sentences)
- AI-generated image
- swipe-based navigation

Users experience the story by swiping vertically or horizontally through the cards.

The system generates the cards automatically using LLMs and image generation models, then caches them for fast delivery.

For the first version, the app **strictly follows the original story progression**.

# 5. AI Story Generation Pipeline

The pipeline converts a story into structured cards.

## Step 1 — Story Metadata Retrieval

Sources:

Books API
Movie API
Anime API

Example flow:

```
User selects book
Backend fetches metadata
Store in DB
```

---

## Step 2 — Story Breakdown

LangChain prompt generates story sections.

Prompt example:

```
You are a narrative summarization system.

Break the story of {STORY_NAME} into 20 chronological moments.

Each moment should include:

1. short title
2. 1-2 sentence narrative
3. detailed visual description for an illustration

Output JSON format:

[
{
section_index:
title:
text:
image_prompt:
}
]
```

---

## Step 3 — Structured Output

LangChain structured output parser ensures valid JSON.

Example schema:

```
section_index: int
title: string
text: string
image_prompt: string
```

---

## Step 4 — Image Prompt Enhancement

Raw prompts are enhanced to improve image quality.

Example transformation:

Input:

```
Harry receives a letter from Hogwarts
```

Enhanced prompt:

```
fantasy cinematic illustration,
young wizard boy receiving magical letter,
owl flying above,
dramatic lighting,
storybook illustration style
```

LangChain chain:

```
StoryBreakdownChain
        ↓
PromptEnhancementChain
```

---

## Step 5 — Image Generation

Worker service generates image.

Pipeline:

```
image_prompt
    ↓
image generation API
    ↓
store in object storage
    ↓
save image_url
```

---

## Step 6 — Story Section Storage

Final section stored:

```
text
image_url
section_index
```

# 6. Preprocessing Strategy

Generating cards at request time would be slow.

Instead we pre-generate content.

## Pre-generation Pipeline

Trigger:

```
New story added
        ↓
enqueue generation job
        ↓
worker processes sections
        ↓
store results
```

---

## Parallel Processing

Sections can generate images in parallel.

Example:

```
20 sections
20 parallel workers
```

Generation time reduces significantly.

---

# 7. Pre-caching Strategy

We cache generated stories.

Cache layers:

Layer 1

Redis

```
story:{id}:sections
```

Layer 2

CDN caching images

Layer 3

Edge API caching

---

## Preloading Strategy

When user opens story:

```
load first 5 sections
prefetch next 10
```

This eliminates UI delay.
